## Context

Pacred has 2 commission-earning role classes from the legacy PHP system:

1. **Interpreter (ล่ามจีน)** — Chinese-fluent staff who negotiate with suppliers + close deals on the Pacred customer's behalf. Per-job/per-order commission tier. PHP definition: `companyType==1 && department==7 && section in (9,10)` OR `companyType==3 && department==2 && section in (2,3)`. PHP source: `pcs-admin/include/pages/withdraw-commission-interpreter/`.

2. **Sales rep** — Pacred sales team earning margin-per-order. PHP gates by **hardcoded whitelist** (`PCS888 / PCS352 / PCS2000 / PCS2678 / PCS4155`). PHP source: `pcs-admin/include/pages/withdraw-commission-sale/`.

Both follow the SAME withdrawal workflow:
- **Accrual** — system computes per-order commission as orders close (monthly accumulation).
- **Withdrawal request** — staff requests payout from accumulated balance, picks bank account, enters amount.
- **Admin approve + slip upload** — accounting reviews, transfers money externally, uploads bank slip to confirm.
- **Mark paid** — status flips to paid; ledger settles.
- **Thai law WHT 15%** — payouts > 5,000 THB attract 15% withholding tax (Revenue Code §50(1)). Pacred withholds + files ภ.ง.ด.53 separately.

Pacred has **partial** sales-rep support via `team_leaders` table + `/admin/sales-payouts` page (per night-4 batch). This spec FINALIZES that + adds the interpreter role.

PHP workflow pattern (confirmed via sampling `withdraw-commission-interpreter.php`):
```
status=1 (pending) → admin uploads bank slip + adminIDUpdate + dateUpdate → status=2 (approved/paid)
fields: ID, title, amount, nameBank, nameUserBank, noUserBank, imagesSlip, withholding, adminIDIP
```

## Data model

### `commission_tiers` — lookup for per-role/per-service commission rate

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `role_kind` | text check | `interpreter` · `sales_rep`. |
| `service_kind` | text check | `service_order` (China shop) · `forwarder` (cargo import) · `freight_quote` (freight conversion). |
| `tier_name` | text | display label, e.g. "interpreter standard rate", "sales VIP closer". |
| `rate_pct` | numeric(6,3) | percentage of base, e.g. 1.500 = 1.5%. |
| `flat_thb` | numeric(12,2) nullable | OR a flat per-job amount (`rate_pct=null` + `flat_thb=200`). |
| `min_base_thb` | numeric(12,2) nullable | min order value to qualify. |
| `effective_from` · `effective_to` | date | tier history (so past accruals don't get re-rated). |
| `is_active` | bool default true | |
| `created_at` · `updated_at` | timestamptz | |

**Constraint:** `commission_tiers_rate_xor_flat`: exactly one of `rate_pct` / `flat_thb` is non-null.

### `commission_accruals` — earned-but-unpaid commission rows

One row per closed order × earning staff. Generated by background job (cron) when the source order reaches a terminal state (delivered / completed).

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `earner_admin_id` | uuid FK → `profiles(id)` | the staff member earning the commission. |
| `role_kind` | text check | `interpreter` · `sales_rep`. |
| `tier_id` | uuid FK → `commission_tiers(id)` on delete restrict | snapshot at accrual time — frozen. |
| `source_kind` | text check | `service_order` · `forwarder` · `freight_quote`. |
| `source_ref` | text | the source order id (`h_no` / `f_no` / `quote_no` depending on `source_kind`). |
| `base_thb` | numeric(12,2) | the base the rate applied to (e.g. order total). |
| `accrued_amount_thb` | numeric(12,2) | computed = `base × tier.rate_pct/100` (or `tier.flat_thb`); frozen at accrual. |
| `accrued_at` | timestamptz | when the source order closed + this accrual was minted. |
| `withdrawal_item_id` | uuid FK → `commission_withdrawal_items(id)` nullable | NULL = unpaid; set when this accrual is included in a paid-out withdrawal. |
| `notes` | text | |
| `created_at` | timestamptz | |

**Indexes:**
- `commission_accruals (earner_admin_id, accrued_at desc) WHERE withdrawal_item_id IS NULL` — fast "my unpaid balance" query.
- `commission_accruals (source_kind, source_ref)` UNIQUE — prevent double-accrual per source × earner (constraint with `earner_admin_id` for compound uniqueness).

### `commission_withdrawals` — withdrawal request header

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `withdrawal_no` | text unique | format: `CW-{YYMM}-{seq}`. |
| `earner_admin_id` | uuid FK → `profiles(id)` | who is being paid. |
| `role_kind` | text check | `interpreter` · `sales_rep`. |
| `title` | text | display label (e.g. "ค่าคอมล่ามจีนเดือนพฤษภาคม 2026"). |
| `gross_thb` | numeric(12,2) | sum of `accrued_amount_thb` for the included `commission_withdrawal_items`. |
| `wht_rate_pct` | numeric(4,2) default `15.00` | Thai WHT for service payments > 5k (per Revenue Code §50(1)) — staff can override to 0 for taxable-elsewhere cases (audited). |
| `wht_amount_thb` | numeric(12,2) | = `round(gross × wht_rate/100, 2)` when gross > 5000 AND wht_rate > 0; else 0. |
| `net_thb` | numeric(12,2) | = `gross - wht_amount`. |
| `payee_bank_name` | text | from selected bank account snapshot. |
| `payee_account_name` | text | |
| `payee_account_no` | text | |
| `status` | text check | `pending` → `approved` → `paid` ; `rejected` (with reason). |
| `requested_at` | timestamptz | |
| `approved_at` · `approved_by_admin_id` | nullable | accounting + super only. |
| `rejected_at` · `rejected_by_admin_id` · `rejected_reason` | nullable | |
| `paid_at` · `paid_by_admin_id` | nullable | when admin clicks "บันทึกการจ่าย" + uploads slip. |
| `slip_storage_path` | text nullable | `commission-slips/{earner_admin_id}/{withdrawal_no}.{ext}`. |
| `notes` | text | |
| `created_at` · `updated_at` | timestamptz | |

**Constraints:**
- `commission_withdrawals_status_chk`: in the enum
- `commission_withdrawals_rejected_consistency`: `status='rejected' → rejected_* all set + rejected_reason ≥ 3 chars`
- `commission_withdrawals_paid_consistency`: `status='paid' → paid_at + paid_by_admin_id + slip_storage_path all not null`
- `commission_withdrawals_wht_consistency`: `wht_amount_thb = 0 OR (gross_thb > 5000 AND wht_rate_pct > 0)` (with manual override audit-flag for off-cycle settlements).

### `commission_withdrawal_items` — links accruals to a withdrawal

Many-to-one: a withdrawal aggregates N accruals.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `commission_withdrawal_id` | uuid FK → `commission_withdrawals(id)` on delete restrict | |
| `commission_accrual_id` | uuid FK → `commission_accruals(id)` on delete restrict UNIQUE | one accrual can be in at most one withdrawal. |
| `included_amount_thb` | numeric(12,2) | snapshot of `accruals.accrued_amount_thb` at request time (frozen even if accrual tier changes later). |
| `created_at` | timestamptz | |

**Index:** `commission_withdrawal_items (commission_withdrawal_id)` for fast aggregation.

### Storage bucket

`commission-slips/` — private, super + accounting write; earner read own (RLS by joining through `commission_withdrawals.earner_admin_id`).

### `admins.role` enum extension

Add `interpreter` to `admins.role` check constraint (currently per [ADR-0002](../decisions/0002-admin-architecture.md): `super | ops | accounting | sales_admin | warehouse | driver`). New full enum:

```
super | ops | accounting | sales_admin | warehouse | driver | interpreter
```

`interpreter` role gets: read own commission accruals + withdrawals + create withdrawal request. NO admin-side access to others' data. Migration adds the role + grants minimal `/profile/commission` portal route access.

### RLS policies

```sql
alter table commission_tiers enable row level security;
alter table commission_accruals enable row level security;
alter table commission_withdrawals enable row level security;
alter table commission_withdrawal_items enable row level security;

-- Earner reads own data
create policy commission_accruals_earner_read on commission_accruals for select
  using (earner_admin_id = auth.uid());
create policy commission_withdrawals_earner_read on commission_withdrawals for select
  using (earner_admin_id = auth.uid());
create policy commission_withdrawal_items_earner_read on commission_withdrawal_items for select
  using (exists (select 1 from commission_withdrawals w
                 where w.id = commission_withdrawal_items.commission_withdrawal_id
                   and w.earner_admin_id = auth.uid()));

-- Earner creates own withdrawal request (status starts pending)
create policy commission_withdrawals_earner_request on commission_withdrawals for insert
  with check (earner_admin_id = auth.uid() and status = 'pending');

-- Admin super + accounting: full read + write
create policy commission_tiers_admin_all on commission_tiers for all
  using (is_admin(array['super','accounting']))
  with check (is_admin(array['super','accounting']));
create policy commission_accruals_admin_all on commission_accruals for all
  using (is_admin(array['super','accounting']))
  with check (is_admin(array['super','accounting']));
create policy commission_withdrawals_admin_all on commission_withdrawals for all
  using (is_admin(array['super','accounting']))
  with check (is_admin(array['super','accounting']));
create policy commission_withdrawal_items_admin_all on commission_withdrawal_items for all
  using (is_admin(array['super','accounting']))
  with check (is_admin(array['super','accounting']));
```

## Background accrual job

`app/api/cron/commission-accrual/route.ts` — runs nightly (or hourly during launch period).

```ts
// Pseudo-code per role
for each terminal-state order created since last_run:
  determine earner_admin_id (from order's owner_rep_id / interpreter_id field)
  determine role_kind + source_kind
  match tier_id from commission_tiers where role_kind + service_kind + active + within effective_from/_to
  compute accrued_amount = base × rate_pct/100 OR flat_thb
  INSERT into commission_accruals (idempotent via UNIQUE (earner_admin_id, source_kind, source_ref))
  audit
```

**Idempotency:** unique constraint on `(earner_admin_id, source_kind, source_ref)` prevents double-accrual. Cron is safe to re-run.

**Open question for ก๊อต:** which `service_orders` / `forwarders` / `freight_quotes` field identifies the earning staff for each order type? Likely:
- `service_orders.adminID_sale` (existing) → sales rep
- `forwarders.adminID_sale` (existing) → sales rep
- NEW: `service_orders.interpreter_admin_id` (need to add) → for orders where interpreter assisted
- NEW: `forwarders.interpreter_admin_id` (need to add)
- `freight_quotes.created_by_admin_id` → sales rep (or specific approver)

Confirm before ภูม implements accrual rules; schema migration may need to add `interpreter_admin_id` columns to existing tables.

## Server actions outline

`actions/admin/commission-tiers.ts` (super + accounting):

```ts
adminUpsertCommissionTier(...)
adminDeactivateTier(id)
```

`actions/admin/commission-withdrawals.ts` (super + accounting):

```ts
adminApproveCommissionWithdrawal(id): Promise<AdminActionResult>
//   pending → approved; stamps approved_at + approved_by

adminRejectCommissionWithdrawal(id, reason): Promise<AdminActionResult>
//   pending → rejected; reason required

adminMarkCommissionPaid(id, slip_file): Promise<AdminActionResult>
//   approved → paid; uploads slip to commission-slips/
//   stamps paid_at + paid_by; notifies earner ('commission_paid')
//   (slip upload happens via FormData → server action)
```

`actions/commission-withdrawals.ts` (earner-self-service via interpreter / sales_admin / sales rep):

```ts
listMyAccruals(): Promise<...>  // shows unpaid + paid history
listMyWithdrawals(): Promise<...>
requestCommissionWithdrawal({
  accrual_ids: string[];  // earner picks which accruals to bundle
  payee_account_id: string;  // from earner's saved bank accounts
  title?: string;
}): Promise<...>
//   validates: all accruals belong to earner_admin_id, all unpaid, sum > 0, payee account exists + owned
//   computes wht_amount per Thai law (gross > 5000 AND wht_rate=15 → withhold)
//   atomic transaction: insert commission_withdrawals + N commission_withdrawal_items + link accruals
//   notifies super + accounting ('commission_withdrawal_requested')
```

**Idempotency:** `commission_withdrawal_items.commission_accrual_id` UNIQUE prevents double-include of an accrual.

**Audit:** every action writes `admin_audit_log` per ADR-0014. Namespace: `commission_tier.*` + `commission_withdrawal.*` + `commission_accrual.*`.

## Acceptance

- Background cron mints accruals from terminal-state orders for both interpreter + sales rep roles; idempotent on re-run.
- Earner can see own unpaid balance + accrual breakdown in portal.
- Earner can bundle N accruals + request withdrawal; UI shows live total + WHT preview.
- Admin sees pending-approval queue; can approve/reject; reject requires reason.
- Admin marks paid + uploads slip; status flips + earner notified.
- Slip stored in private `commission-slips/` bucket; earner can download own.
- WHT computed correctly per Thai law (15% on gross > 5,000 THB; can override + audit).
- Monthly report sums WHT withheld for ภ.ง.ด.53 filing.
- `interpreter` role added to RBAC; interpreter staff have minimal access (own commission portal only).
- Every action audited per ADR-0014; double-accrual structurally impossible via UNIQUE constraint.
