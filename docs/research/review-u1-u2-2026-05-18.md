## Verdict

**🟡 SHIP-WITH-FOLLOW-UP — but P0-1 must be fixed before the next production deploy.**

The U1/U2 code is structurally sound: idempotency guards are present and well-reasoned,
status-machine race-guards (`.eq("status", expectedFrom)`) are consistent, RLS is enabled
on every new table, the compensating-delete on `adminMarkRefundPaid` is correct, and the
`0066` terminal-reversal trigger genuinely closes the double-credit hole the U1 audit
flagged. The freight auto-chain wiring is best-effort and cannot roll back its parent.

The blocker is **not a code bug but a missing business rule**: the refund money path
(U1-6) will mark-paid and credit a customer's wallet for **any amount, against any order,
with no check that the order was ever paid and no cap against what was actually
collected**. With migrated PCS customers (U2-1) about to flood in, this is an exploitable
money-loss surface. It needs an amount/paid-status guard before deploy.

One **P1 IDOR** (admin-created refund on a non-owned parent) and several P1/P2 correctness
issues follow. Counts: **1 P0 · 5 P1 · 7 P2**.

## Detail

### P0-1 — Refund path moves money with no amount cap and no paid-status check  · **for ภูม**

**Files:** `actions/admin/refunds.ts:227` (`adminMarkRefundPaid`), `actions/refunds.ts:38`
(`customerCreateRefundRequest`), `actions/admin/refunds.ts:58` (`adminCreateRefund`),
migration `0058_refund_requests.sql`.

**What's wrong.** The refund money path validates only that:

- `amount_thb > 0` and `≤ 9,999,999.99` (Zod, `lib/validators/refund.ts:59`; DB CHECK
  `0058:69`);
- the parent order **exists** (and, customer-side, is RLS-visible — i.e. owned).

It **never** checks:

1. **that the parent order was ever paid.** A customer can create a forwarder, leave it
   at `pending_payment` (never pay a baht), then immediately file a refund request
   against its `f_no`. The order-existence probe in `customerCreateRefundRequest`
   (`actions/refunds.ts:55-76`) succeeds — RLS only checks ownership, not status.
2. **that the refund amount is ≤ what the customer actually paid.** `adminMarkRefundPaid`
   credits `Number(row.amount_thb)` straight into the wallet (`actions/admin/refunds.ts:267`).
   Nothing reconciles it against `forwarders.total_price`, the `wallet_transactions`
   `import_payment` debit sum, or `service_orders.total_thb`. A customer who paid ฿500
   can request — and an inattentive `accounting` admin can mark-paid — a ฿50,000 refund.
3. **partial-refund accumulation.** `0058`'s header explicitly designs partial refunds as
   "customer creates a new request for the remainder." Nothing sums prior `paid` refunds
   for the same `source_ref` — so N requests of the full amount each can all be paid.

The mark-paid credit is `kind='refund'`, positive amount; the `0064` overdraw guard
**ignores credits by design**. So there is no DB-level backstop either — the wallet just
goes up.

**Why it matters.** This is a direct, unbounded money-loss path. The grep confirms the
only `paid` references in `actions/refunds.ts` + `actions/admin/refunds.ts` are the
refund's *own* status — never the parent order's. `gap-revenue-flow.md` H-3 framed U1-6 as
"status without money path"; the fix introduced the money path but not the guard rails.
With ~9,000 PCS customers migrating in this week (U2-1), the blast radius is the whole
book. The admin UI shows the parent's `total_price` on the detail page
(`admin/refunds/[id]/page.tsx:104`) — so a *careful* admin can eyeball it — but "careful
admin" is not a control.

**Suggested fix (before next deploy).** Add a server-side ceiling in
`adminMarkRefundPaid` (and ideally at request creation):

- resolve the parent's collected amount — for `forwarder`: sum `wallet_transactions`
  where `reference_type='forwarder' AND reference_id=f_no AND kind='import_payment' AND
  status='completed'`; for `service_order`: the `order_payment` slice; for
  `yuan_payment`: the transfer's THB amount;
- sum already-`paid` `refund_requests` for the same `source_ref`;
- reject mark-paid if `amount_thb + prior_refunds > collected`. For `source='manual'`
  (no parent) keep it admin-judgement but log loudly.
- Reject refund *creation* against a parent in a never-paid status
  (`pending_payment` / `awaiting_payment` / `pending`).

A DB CHECK can't express this (cross-table) — it must be an action-layer guard, mirroring
the `billing-gate.ts` pattern.

### P1-1 — `adminCreateRefund` does not verify the parent belongs to the named customer (IDOR) · **for ภูม**

**File:** `actions/admin/refunds.ts:341` (`verifySourceRef`), called from `:82`.

`adminCreateRefund` takes a `profile_id` (the customer to refund) and a `source_ref`.
`verifySourceRef` looks up `forwarders.f_no` / `service_orders.h_no` / `yuan_payments.id`
and confirms the row **exists** — but never checks `forwarder.profile_id === d.profile_id`.
Contrast `adminCreateShipmentManual` (`actions/admin/warehouse.ts:572-588`) which *does*
do exactly this `profile_id` cross-check.

**Why it matters.** An admin (or a `sales_admin`/`ops` whose account is compromised, or a
simple fat-finger) can create a refund crediting customer B's wallet while citing
customer A's order as justification. The audit trail then shows a coherent-looking
`source_ref` that points at the wrong customer — making the fraud or the mistake hard to
spot in reconciliation. Combined with P0-1 (no amount cap) this is an internal-fraud
primitive.

**Suggested fix.** In `verifySourceRef`, also select `profile_id` and return an error if
it ≠ the refund's target `profile_id`. (`yuan_payments` likewise has a `profile_id`.)

### P1-2 — PCS backfill `createUser` collides on existing phone/email · **for ภูม**

**File:** `actions/admin/pcs-migration.ts:280` (`admin.auth.admin.createUser`).

The backfill calls `createUser({ phone, email, ... })` with no pre-check that the phone or
email is free. Supabase rejects a duplicate phone/email with a unique violation → the row
lands in `result.failed` with the raw error. Two real scenarios:

1. The same person already signed up natively on Pacred (phone reused) — their migration
   row silently fails; they end up with a native profile *and* a stuck staging row.
2. The legacy `tb_users` dump itself has duplicate phones (very common in 9,000 rows of
   8-year-old data) — the *second* occurrence fails.

The loop continues (good — one bad row doesn't block 9,000), but the failures are only
visible by reading `errors[]`. There's no merge/relink path.

**Why it matters.** Launch-week migration of the entire customer base; a meaningful
fraction will fail this way and those customers cannot log in to the rebranded system —
the exact "customers drop + เสียชื่อ" outcome the emergency sprint exists to prevent.

**Suggested fix.** Before `createUser`, look up an existing `auth.users`/`profiles` by
normalized phone and by email. On hit: link the staging row to the existing profile
(set `legacy_pcs_user_id` + `migrated_from_pcs` on it, record in `notes`), don't create a
duplicate. De-dupe the staging table on phone before the run, or pick a deterministic
winner. This is the same "already_in_profiles" skip pattern, extended to phone/email keys.

### P1-3 — Billing gate fails OPEN on DB read error · **for ภูม**

**File:** `lib/forwarder/billing-gate.ts:96` and `:120`.

The gate's header documents the fail-open choice deliberately ("let the underlying
wallet-tx insert decide"). For the **forwarder** read that's defensible — a missing
forwarder means the caller's own `not_found` guard fires. But for the **`cargo_containers`
read** (`:113-120`), fail-open means: forwarder is `arrived_thailand`, container *is*
linked, but the container-status read hits a transient error → gate returns
`{ blocked:false }` → the post-arrival wallet debit proceeds on the **order-time CBM
estimate**. That is precisely the ~31% over/under-bill U1-3 was built to prevent
(datanew L-3).

The downstream `wallet_transactions` insert has no idea CBM reconciliation was skipped —
it succeeds. There is no synthetic error to surface; the money just moves on stale data.

**Why it matters.** A gate that fails open on the error path is not a gate for the case
that matters. The "DB outage surfaces downstream" rationale holds for a *hard* outage but
not for a *transient* blip on one read — and a transient blip is the common case.

**Suggested fix.** Fail **closed** on the `cargo_containers` read error specifically
(return `awaiting_container_close` with an `error`-ish reason), the same way the code
*already* fails closed when the container row has vanished (`:123`). The forwarder-read
fail-open can stay. Note `billing-gate.test.ts` cases (g) assert fail-open for both — that
test would need updating alongside the fix (flag to ภูม; do not edit here).

### P1-4 — `0067` migration's "idempotent backfill into profiles" is a comment, not code · **for ภูม**

**File:** `0067_pcs_customer_migration.sql` lines 105-130.

The migration header (lines 30-33) and the section banner ("3) Idempotent backfill into
profiles") promise a backfill step. The body is **only a comment** explaining why it's a
no-op and deferring everything to the server action. That's an *acceptable* design — but
the migration *documents itself as doing something it does not do*. Anyone reading
`0067` (or the runbook) to verify "did the backfill run" will be misled: `0067` applying
cleanly proves nothing about customer data.

**Why it matters.** Launch-week migration with a runbook; a doc/code mismatch on a
data-migration step is how a step gets skipped. The `v_pcs_migration_status` view is the
real verification surface — the migration should say so unambiguously and the banner
should read "(intentionally no-op — see server action)".

**Suggested fix.** Either delete the misleading "3) Idempotent backfill" banner, or
demote it to an explicit `-- (NO-OP — backfill is adminBackfillPcsAuthUsers())`. Pure
doc/comment change; no behavior change. (ภูม owns this file + the runbook.)

### P1-5 — Container→order status cascade is best-effort and non-atomic · **for ภูม**

**File:** `actions/admin/warehouse.ts:154` (`cascadeContainerToShipments`) +
`:203` (`cascadeShipmentToOrders`), invoked from `adminSetContainerStatus:417`.

When an admin flips a container to e.g. `arrived`, the code cascades container → each
shipment → each forwarder/service_order. Every hop is wrapped in try/catch and **logs +
continues** on failure (documented "best-effort"). So a partial failure is possible:
container = `arrived`, shipment A's forwarder updated to `arrived_thailand`, shipment B's
forwarder update threw → B's forwarder stuck at `in_transit`.

**Why it matters.** The billing gate (`billing-gate.ts`) keys off `forwarder.status ∈
{arrived_thailand, out_for_delivery}`. A forwarder that *should* be `arrived_thailand` but
is stuck `in_transit` will **not** be gated — `getCargoBillingGate` returns
`{blocked:false}` for `in_transit` (`billing-gate.ts:105`) — so a wallet debit can be
billed on the stale estimate. The cascade's non-atomicity quietly defeats P1-3's sibling
control. It also produces customer-facing "where is my cargo" divergence (the OT-* leak
holes in the chat decode).

**Suggested fix.** Best-effort cascade is a reasonable V1 choice, but it needs a
reconciliation backstop: a periodic job (or an admin "re-sync container" button) that
re-derives child statuses from the container, and surfacing partial-cascade failures in
the admin UI rather than only in `logger.error`. At minimum, count failures and return
them in the action result so the admin sees "3 of 12 children did not update."

### P2-1 — Refund serial burned on failed INSERT · **for เดฟ**

`next_refund_request_no()` (`0058:135`) increments the daily counter and returns the
serial; the caller then INSERTs. If the INSERT fails (constraint, race), the serial is
already consumed → the `RF-YYMMDD-NNNN` sequence has a hole. Same pattern as the freight
quote/invoice serials (a known accepted trade-off elsewhere, see
`freight-invoices.ts:424` "gap will be logged"). Non-blocking; note for consistency.
Fix: reserve the serial *inside* the same statement as the INSERT, or accept + document
the gap as the freight code does.

### P2-2 — `0059` container-unify backfill aborts on code collision · **for ภูม**

`0059` lines 158-166 acknowledge that a legacy `CN-` code colliding with an existing
spine `code` would make the `INSERT` raise and "caller errors out loud." But the
migration is declared "Safe to re-run" — and on re-run it will hit the **same** collision
and abort again, every time, never completing. The `WHERE NOT EXISTS` only covers the
`legacy_container_id` key, not `code`. Fix: pre-detect colliding codes and remap them
(`LEGACY-` prefix) so the backfill is genuinely idempotent, or `ON CONFLICT (code) DO
NOTHING` with a post-run warning count.

### P2-3 — `attachShipmentToSack` does not enforce same-container · **for เดฟ**

`lib/warehouse/sacks.ts:138` documents that it deliberately skips the same-container
check "to allow inter-container moves the MOMO sync occasionally needs." Acceptable for an
automated sync, but if this helper is ever wired to an admin button, a wrong `sackId`
silently mis-files a customer's parcel into a sack inside a *different* container — a
physical-tracking error. Fix: when called from staff UI, pass a `requireSameContainer`
flag; keep the loose path for the sync only.

### P2-4 — Orphaned `auth.users` cleanup is fire-and-forget · **for ภูม**

`actions/admin/pcs-migration.ts:341` — on profile-insert failure the orphan auth user is
removed via `.deleteUser(userId).catch(() => {})`. If that delete itself fails (network,
rate limit), a login-capable `auth.users` row exists with no `profiles` row and **no
trace** — the swallow drops the error entirely. A customer could authenticate into a
broken half-state. Fix: on delete failure, write the `userId` into the staging row's
`notes` (and/or `logAdminAction`) so it's recoverable, instead of `.catch(()=>{})`.

### P2-5 — `cargo_sacks` customer-read RLS leaks other customers' sacks · **for เดฟ**

`0068` lines 154-167: the `cargo_sacks_customer_read` policy's second `OR` branch grants
read to any customer owning ≥1 shipment in the **parent container**. A container holds
many customers' shipments; this means customer A — who has one parcel in container X —
can read **every sack row in container X**, including sacks that contain only customer B's
and C's parcels (sack `code`, `weight_kg`, `cbm`, `note`). Low severity (sack metadata
isn't money/PII) but it's a cross-customer data exposure that contradicts the
shipment-scoped model. Fix: drop the second `OR` branch — a customer should see a sack
only if they own a shipment **in that sack** (the first branch).

### P2-6 — `adminMarkRefundPaid` idempotent replay returns empty `paid_at` · **for เดฟ**

`actions/admin/refunds.ts:247-256` — when the refund is already `paid`, the early-return
gives `{ paid_at: "", wallet_tx_id: row.paid_wallet_tx_id ?? "" }`. The empty-string
`paid_at` is a falsy lie — a caller that does `if (res.data.paid_at)` or renders it gets
wrong behavior on the second call. Cheap fix: `SELECT paid_at` in the same read and return
the real value.

### P2-7 — All migrated PCS customers inserted as `status:"active"` · **for ภูม**

`actions/admin/pcs-migration.ts:333` hard-codes `status:"active"` ("migrated customers are
pre-approved") while `:330` maps `is_active` from legacy `userActive`. So a legacy account
that was *disabled* (`userActive ≠ '1'`) still becomes a live, `active` Pacred login (just
with `is_active=false`). Depending on how the login guard reads these two columns, a
deactivated legacy customer may regain access. Fix: derive `status` from legacy
`userActive` too, or confirm with the team that "all legacy customers re-activate" is
intended (the L-2 chat says sales *phones* them — implying yes — but make it explicit).

## What's correct and well-built (so ภูม knows what NOT to change)

- **`adminMarkRefundPaid` compensating delete** (`refunds.ts:294-312`): wallet credit
  written first, refund-row flip second, orphan wallet tx deleted on flip failure with a
  loud audit row if the compensate *also* fails. Correct ordering, honest error message.
- **`0066` terminal-reversal trigger**: genuinely closes the "re-flip `paid`→`approved`
  then double-credit" hole; `errcode 23514` is the right choice.
- **Idempotency guards** are consistently applied and correct: `wallet_tx_import_payment_uniq`
  (forwarder pay), `wallet_tx_freight_payment_uniq` (freight wallet debit),
  `freight_payment_bank_ref_uniq`, `source_quote_id` UNIQUE (quote→shipment convert),
  `freight_invoices_one_active_per_shipment_uidx` (`0066`) — each paired with a 23505
  catch + re-SELECT for idempotent retry.
- **Status-machine race-guards**: every flip uses `.eq("status", expectedFrom)` as an
  optimistic lock — consistent across refunds, freight shipments, freight quotes.
- **`debitWalletForFreightPayment` auto-void** (`freight-invoice-payments.ts:329-340`):
  if the wallet debit fails, the freight payment row is voided so the invoice never flips
  to `paid` without the money — correct, and the void path reverses the debit too.
- **`0058` customer INSERT RLS** is tight: pins `profile_id=auth.uid()`,
  `status='pending'`, `source` in the non-manual set, and **every** admin-side column
  forced `NULL` — a customer cannot forge a pre-approved/pre-paid row.
- **`0062` role-pin is respected**: new admin RLS (`0058` refund, `0068` sacks) uses
  explicit `is_admin(array[...])` role arrays — no bare `is_admin()` regression. Refund
  write is `['super','accounting']`; sacks write `['super','ops','warehouse']`.
- **Indexes** match the query patterns: `refund_requests` has profile/status/source/no
  indexes; `cargo_sacks` has container + source-updated; FK columns on `forwarders`/
  `service_orders`/`cargo_shipments` are all partial-indexed `WHERE ... IS NOT NULL`.
- **All five migrations are idempotent** (`create ... if not exists`, `drop ... if exists`,
  `create or replace`) and additive — safe to apply on a live prod DB.

## Recommended deploy gate

1. **P0-1 must be fixed** (refund amount cap + parent-paid check) before the next
   `dave → main` deploy. It is an unbounded money-loss path and the PCS migration
   multiplies its blast radius.
2. **P1-1** (refund IDOR) should ship in the same change — it's a two-line addition to
   `verifySourceRef` and compounds P0-1.
3. **P1-2** (PCS migration phone/email collision) must be fixed before the U2-1 backfill
   is *run* — it does not block the deploy itself, but running the backfill without it
   will strand a chunk of customers.
4. P1-3 / P1-4 / P1-5 and the P2s are follow-ups — not deploy blockers, but P1-3 and P1-5
   together leave a real hole in the U1-3 billing-gate guarantee and should land soon.

**Confidence:** Medium-high in this assessment. The review is static (read-only, no DB,
no run). The money-flow findings were cross-checked by grep against the actual
`paid`/amount references; the RLS findings against the migration SQL directly. P0-1 is
high-confidence. The U1/U2 code is **not yet safe for the next production deploy** — it
becomes safe once P0-1 + P1-1 land.
