## Overview

### Legacy scope (what PCS Cargo's ฝากสั่งซื้อ actually does)

The China shop-order flow is a **cart → order → multi-status-fulfilment → auto-spawn-import** pipeline, all on `tb_cart` → `tb_header_order` (header) + `tb_order` (line items):

1. **Add to cart** → `tb_cart` (4 add modes: paste-URL search, manual custom, admin-on-behalf, bulk). Cart cap = 150 (`countFor = 151 - countCart`).
2. **Cart / checkout** (`cart.php`) → pick address (`tb_address`) + Thai courier (`api-shipBy.php`, 50+ zip/province-gated couriers + VIP/freedom/non-50 JSON overrides) + transport mode (รถ EK / เรือ SEA) + crate. Submit `addOrder`.
3. **Order create** (`shops.php` `addOrder` L4-159): `hNo = 'P' + (max ID + 1)`; INSERT `tb_header_order` (`hstatus='1'`); loop selected cart rows → INSERT `tb_order` + DELETE `tb_cart`; rollup `hTotalPriceCHN`/`hRate`(`tb_settings.rsDefault`)/`hCount`/`hTitle`/`hCover`; assign `adminIDIP` (round-robin ฝ่ายฝากสั่งซื้อ); send email (`contentMailShopNew`); `saveHistory(27)`.
4. **Status fulfilment** (admin `shops.php` update handlers) — the workflow loop:
   - `update2` → **hStatus 2** (รอชำระเงิน): set per-item cPrice/cShippingCHN, recalc, `hDatePayment = NOW()+5d`, email+**SMS**+**LINE**+**LINE-OA** payment-link push.
   - `update_hStatus` → **hStatus 3** (สั่งสินค้า): manual.
   - `update3` → **hStatus 4** (รอร้านจีนจัดส่ง): set cShippingNumber + cPriceUpdate per shop, notify.
   - `saveTarcking`/`arrSaveTarcking` → when every `cShippingNumber` has a matching `cTrackingNumber` → **hStatus 5** (สำเร็จ); **AND auto-INSERT `tb_forwarder` import record** (the cargo→import bridge) + tb_promotion link + forwarder notify.
   - `cancelOrder.php` → **hStatus 6** (ยกเลิก) — guard `hStatus<3`.
5. **Customer pays** → wallet debit (`tb_wallet` + `tb_wallet_hs` type) keyed on `refOrder = hNo`; status flips toward complete.
6. **Print** (`printShop.php`, mPDF THSarabunNew): `print=1` → ใบเสร็จ (hStatus=5, sets `hPrintBill=1`); `print≠1` → ใบแจ้งหนี้ (hStatus 2-5, sets `hPrintBill2=1`); juristic → tb_corporate name/number/address.
7. **Notes** (`saveNote`): `hNoteUser` flag (1=admin-only / 2=customer+admin) → LINE-OA push + unread badge.
8. **Cancel-multiple modal** (`shops/getList.php`): customer selects orders → confirm bulk cancel (`orderCancelAll`).

**Status enum (`tb_header_order.hstatus`) — canonical:** `1`=รอดำเนินการ · `2`=รอชำระเงิน · `3`=สั่งสินค้า · `4`=รอร้านจีนจัดส่ง · `5`=สำเร็จ · `6`=ยกเลิก. **Shop-order cancel = `6`, NOT `99`** (forwarder/yuan use `99`; shop uses `6` — verified in both `cancelOrder.php` L8 and the status-tab switch `shops.php` L256-264).

### Pacred scope (what exists)

Pacred has built **TWO parallel, non-reconciling customer shop systems**:

| | **Path A — ปอน faithful `/cart/`** | **Path B — rebuilt `/service-order/cart/` (LIVE)** |
|---|---|---|
| Cart store | `tb_cart` (read/calc/delete/qty only) | **`cart_items`** (rebuilt, empty on prod) |
| Add to cart | none on customer side¹ | `addCartItem` → **`cart_items`** |
| Cart page | `(protected)/cart/page.tsx` (reads tb_cart) | `(protected)/service-order/cart/page.tsx` (CartManager, reads cart_items) |
| Checkout action | `submitCartOrder` → **`tb_header_order` + `tb_order`** ✅ | `placeServiceOrder` → **`service_orders` + `service_order_items`** ❌ |
| Nav linked? | **NO — orphaned** | **YES** (sidebar L76-79 + 9-icon launchpad + cart-badge) |

¹ App-wide, the **only INSERT into `tb_cart` is `actions/admin/cart.ts:133`** (admin-on-behalf). No customer surface ever writes `tb_cart`. So Path A's faithful cart page is fed nothing.

The **read side is correct**: `listServiceOrders` / `getServiceOrder` / `getServiceOrderForReceipt` + the print page all read `tb_header_order` + `tb_order` → a migrated customer **sees** their real orders. But every **write** (create / pay / cancel) lands in rebuilt empty tables. Net effect: a customer can browse historical orders but **cannot place, pay, or cancel any order through the live UI** — the worst silent-dead-write surface in the whole port.

### % complete

**~20%.** Read-side (list/detail/receipt/print, tb_header_order) ✅ + ปอน's orphaned faithful checkout exists ✅. But the **entire live write path is dead** (cart_items + service_orders + wallet_transactions, all rebuilt), the two carts never reconcile, and the legacy multi-status admin loop + notify + auto-import-spawn are unwired on the customer-touching side. Matches ภูม's master-fidelity number (service-orders ~15-25%).

## Death-flows (P0/P1 detailed)

### 💀 P0-1 — Customer cannot PLACE an order (live checkout writes dead `service_orders`)
- **Live path:** sidebar/launchpad → `/service-order/add` → `addCartItem` → `cart_items` → `/service-order/cart` → `CartManager` → `placeServiceOrder` (`actions/service-order.ts:606-754`) → INSERT `service_orders` + `service_order_items` (rebuilt, empty on prod).
- **Legacy spec:** `shops.php` `addOrder` → INSERT `tb_header_order` (hno='P'+id, hstatus='1') + `tb_order`.
- **Effect:** a new order is invisible to admin (admin reads tb_header_order) AND to the customer's own list (`listServiceOrders` reads tb_header_order). Revenue order silently lost. **The fix already exists** — `submitCartOrder` (`actions/cart.ts:228`) is a faithful tb_header_order transcription, but is wired only to the orphaned `/cart/` page.
- **Owner:** เดฟ (routing + schema unification — architecture).

### 💀 P0-2 — Customer cannot CANCEL a real order
- `cancelServiceOrder` (`actions/service-order.ts:759`) → `UPDATE service_orders SET status='cancelled' WHERE h_no=$hNo`. Real orders live in `tb_header_order` → 0 rows match → silent no-op. UI shows success.
- **Legacy:** `cancelOrder.php` → `UPDATE tb_header_order SET hStatus='6' WHERE hStatus<3 AND hNo=…`. Two bugs vs legacy: (a) wrong table; (b) value `'cancelled'` not `'6'`; (c) missing the `hStatus<3` guard (legacy only lets you cancel before "สั่งสินค้า").
- **Owner:** เดฟ.

### 💀 P0-3 — Pay-from-wallet fails on real orders + targets wrong wallet
- `payServiceOrderFromWallet` (`actions/service-order.ts:800`) → `SELECT FROM service_orders WHERE h_no=$hNo` → not_found for real orders. Even if found, debits `wallet_transactions` (rebuilt) keyed `order_header`, NOT `tb_wallet`/`tb_wallet_hs` where the 8,898 customers' real balances live.
- **Legacy:** wallet debit on `tb_wallet` + `tb_wallet_hs` with `refOrder=hNo`; status guard against double-pay (`update2` checks `tb_wallet_hs status IN (1,2) AND refOrder=hNo`).
- **Effect:** customer self-pay is completely broken for migrated orders; doubles down on the rebuilt-wallet gap (master-audit Pattern 3).
- **Owner:** เดฟ (depends on P0-1 unifying the order table + the wallet-ledger decision).

### 💀 P0-4 — Split-brain cart: add → `cart_items`, faithful cart reads `tb_cart`
- App-wide, the only `tb_cart` INSERT is `actions/admin/cart.ts:133`. Customer `addCartItem`/`addCartItemsBulk` → `cart_items`. So even if `/cart/` (faithful) were re-linked, it would show an empty cart because nothing populates tb_cart on the customer side.
- **Fix shape:** pick ONE store. Recommended: make customer add-to-cart write `tb_cart` (re-point `addCartItem` + the `/search` add UI), delete the `cart_items` reads, re-link nav to `/cart/` + `submitCartOrder`. Then delete `placeServiceOrder` + `/service-order/cart/CartManager` + `cart_items` table.
- **Owner:** เดฟ.

### 🟠 P1-5 — Admin status loop + auto-spawn `tb_forwarder` (cargo→import bridge)
- Legacy `saveTarcking` auto-creates a `tb_forwarder` row when an order completes (every shop order that ships becomes an import record — this is how ฝากสั่งซื้อ feeds ฝากนำเข้า). Pacred `service-orders-spawn.ts` exists but writes to the rebuilt path (per ภูม's master-audit) — needs verification it targets `tb_forwarder`.
- This is the **admin back-office** lane → **owner: ภูม**. Listed here because the customer's "สำเร็จ" status + the import record they later track both depend on it.

### 🟡 P1-6 — Print: hPrintBill flag-set + juristic branch unverified
- `/service-order/print` + `/api/pdf/shop-order/[hNo]` read `tb_header_order` ✅, and the page comment claims `print=1 → UPDATE hPrintBill='1'` / `print≠1 → hPrintBill2='1'`. Verify the flag UPDATE actually fires and the juristic (tb_corporate) name/number/address branch (`printShop.php` L57-82) is reproduced. **Owner:** ปอน (customer-facing print surface).

## Flow-order divergences

1. **Cancel status value** — legacy `tb_header_order.hStatus='6'`; Pacred `service_orders.status='cancelled'` (string). Even after pivoting to tb_header_order, must write `'6'` (the 1-char code), not a word. (Re-confirms the audit caveat: shop cancel is `6`, the forwarder/yuan `99` does NOT apply here.)
2. **Cancel guard** — legacy only allows cancel while `hStatus<3` (before สั่งสินค้า). Pacred `cancelServiceOrder` allows `["pending","awaiting_payment"]` on the rebuilt enum — needs mapping to `hstatus IN ('1','2')`.
3. **Order-create status** — legacy initial `hstatus='1'` (รอดำเนินการ) and admin moves to `2` (รอชำระเงิน) only after pricing (`update2`). Pacred `placeServiceOrder` jumps straight to `status='awaiting_payment'` (= legacy 2) at create time — **skips the รอดำเนินการ (1) review step**. The faithful `submitCartOrder` correctly sets `hstatus='1'`. Wiring the faithful path fixes this automatically.
4. **Wallet-pay double-pay guard** — legacy blocks re-pay by checking `tb_wallet_hs status IN (1,2)`; Pacred uses a `wallet_transactions` partial-unique index on `order_header` — different mechanism on a different (rebuilt) table; will need re-anchoring when pay moves onto the legacy wallet.
5. **Provider→shop grouping order** — legacy groups DISTINCT cProvider then DISTINCT cNameShop in result order with a running `noRow`; the faithful `/cart` reproduces this; the rebuilt CartManager groups differently. Cosmetic, resolves with unification.
