# บทเรียน: owner เปลี่ยนกฎเงินกลางวันเดียวกัน — ต้องตามแก้ทั้ง "โค้ด" และ "ข้อมูลที่ backfill ไปแล้ว"

**วันที่:** 2026-07-21 · **บริบท:** กติกาเก็บเงินค่าขนส่งในไทย (ต้นทาง/ปลายทาง)

## เกิดอะไรขึ้น

**เช้า** owner สั่งผม (เดฟ):
> "ถ้าเรื่องขนส่งเอกชน ต้องตีเป็นชำระปลายทางทั้งหมดนะครับ แล้วพอเลือกชำระปลายทาง ก็ต้องไม่ใส่ค่าขนส่งไทย ค่าขนส่งไทยก็ควรเป็น 0"

ผมล็อกกติกาในโค้ด (SOT `enforceCodDomesticZero`) + wire 11 write-path + **backfill prod** (ล้างค่าส่งไทยของแถว COD · ตั้งขนส่งเอกชน = ปลายทาง).

**บ่ายวันเดียวกัน** owner สั่ง ภูม ละเอียดกว่า:
> "5 รายการที่เก็บเงินเป็น ต้นทาง ถ้าเลือก: Flash · J&T · ไปรษณีย์ไทย · เหมาๆ · Pacred express. ที่เหลือ ต่างจังหวัด ขนส่งเอกชนทั้งหมด เก็บเป็นปลายทาง. ให้ล็อคไปเลย."

= **3 เจ้าประจำชาติที่ Pacred จ่ายเงินก่อน ย้ายกลับเป็นต้นทาง** — ขัดกับกฎเช้าตรงๆ บนเส้นเงิน

## กับดัก

โค้ดของ ภูม ถูกและ merge ได้ทันที (เขา re-point จุดที่ผมวางไว้ให้ตรง SOT ใหม่ครบด้วย) — **แต่ข้อมูลที่ผม backfill ไปตามกฎเช้ายังค้างอยู่บน prod**:
- 9 แถว Flash ถูกตั้งเป็น "ปลายทาง" + ค่าส่งไทย 0 ตามกฎเช้า
- กฎบ่ายบอกว่า Flash = ต้นทาง → 9 แถวนี้จะไม่มีใครเก็บค่าส่งไทย และไม่มีอะไรมาแก้ให้เอง

**merge โค้ดอย่างเดียว = กติกาถูก แต่ข้อมูลผิดเงียบๆ** — ไม่มี error, ไม่มีเทสแดง, เห็นตอนเก็บเงินขาดเท่านั้น

## สิ่งที่ต้องทำทุกครั้ง (checklist)

1. **อ่าน quote ของ teammate ก่อนตัดสิน** — ถ้าเขาบันทึก owner-quote ที่ใหม่กว่าและละเอียดกว่า ให้ยึดอันใหม่ (อย่าถือว่าของเรา "มาก่อน" แปลว่าถูกกว่า)
2. **ตรวจว่าเขา re-point จุดที่เราวางไว้ครบไหม** — SOT เปลี่ยนความหมาย (เช่น `isOwnFleetCarrier` → `isPayAtOriginCarrier`) แต่ caller เก่ายังเรียกตัวเดิม = พฤติกรรมครึ่งๆ
3. **ไล่ DATA ที่เรา migrate ไปตามกฎเก่า** — เขียนสคริปต์แก้กลับ (dry-run + backup + txn) และใช้ backup ของรอบก่อนคืนค่าที่ถูกล้างไป
4. **verify ด้วย query "ผิดกติกา = 0"** ไม่ใช่ดูว่าโค้ดคอมไพล์ผ่าน — เช่น 3 ข้อ: (ก) ขนส่งต้นทางแต่ยังเป็นปลายทาง = 0 (ข) เจ้าอื่นแต่ยังเป็นต้นทาง = 0 (ค) ปลายทางแต่ยังมีค่าส่งไทย = 0
5. **บันทึกวิวัฒนาการของกฎในความจำ** พร้อมวันเวลา — กฎที่เปลี่ยนกลางวันเดียวกันคือกับดักของ session ถัดไป

## ของแถมที่เจอจากเคสนี้

- **แก้ราคาแล้วต้องไล่ "กฎที่บล็อก" ด้วย** — ตอน owner สั่ง "Pacred Express ไม่ต้องคิดขั้นต่ำ ใส่ 0 ได้" ถ้าแก้แค่สูตรราคา พนักงานจะใส่ 0 ได้จริงแต่ **วางบิลไม่ได้** เพราะกฎ "ห้ามลืมค่าส่งไทย" ยังมองว่า 0 = ลืมกรอก → ต้องปลดคู่กันเสมอ
- **การล้างค่าที่ระบบอื่นใช้อยู่** — ค่าส่งไทยถูกใช้เป็น "ยอดเก็บปลายทาง" ในไฟล์ CSV ที่ส่งให้ Flash ด้วย · ล้างเป็น 0 ดื้อๆ = ขนส่งไม่เก็บเงินลูกค้า → ต้องหา fallback (คิดสดจากเรทเดิม) ก่อนล้าง
