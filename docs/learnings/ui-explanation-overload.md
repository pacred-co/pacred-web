# คำอธิบายเยอะ = ดูเหมือนงานยังไม่เสร็จ (owner 2026-07-23)

## สิ่งที่ owner พูด (verbatim)

> "พวกคำอธิบาย เรามองว่ามันยังเยอะเกินไปอยู่เลยครับ ไล่คลีนออกให้เลยครับ
> **user ทั่วไปไม่กล้าใช้หรอกครับ เหมือนบัค เหมือนงานยังไม่เสร็จเลยครับ ไม่มีใครกล้าใช้หรอกครับ**"

## ทำไมมันสวนสัญชาตญาณคนเขียนโค้ด

เราคิดว่า "อธิบายเยอะ = ใจดี = ลดคำถาม" แต่ผู้ใช้จริงอ่านออกมาเป็นอีกอย่าง:

| เราตั้งใจ | ผู้ใช้อ่านได้ |
|---|---|
| กันเข้าใจผิด | "เรื่องนี้มันพังบ่อยแน่ๆ เขาถึงต้องมาเตือน" |
| บอกข้อยกเว้นให้ครบ | "ยังทำไม่เสร็จ เลยต้องมาเขียนแก้ตัว" |
| โปร่งใส | "ซับซ้อนขนาดนี้ กดผิดแล้วซวยแน่" → **ไม่กล้ากด** |

หน้าที่ต้องมีกล่องเตือนล้อมรอบ ส่งสัญญาณว่า **ตัวหน้ายังออกแบบไม่จบ** — ไม่ใช่ว่าเอกสารดี.

## กติกาที่ใช้ต่อจากนี้

1. **คำอธิบายที่ยาวกว่าตัวงาน = สัญญาณให้ไปแก้ตัวงาน** ไม่ใช่แก้คำอธิบาย.
   ถ้าต้องเขียน 6 บรรทัดบอกวิธีใช้ปุ่ม → ปุ่มนั้นวางผิดที่ หรือ flow ผิด
   (รอบนี้: ย้ายปุ่ม "บันทึกต้นทุน" ลงใต้สรุป + ติดป้าย "ขั้นที่ 1/2" แล้วคำอธิบายหายไปเอง)
2. **เอา step ใส่ในตัว UI แทนใส่ในย่อหน้า** — ป้าย "ขั้นที่ 1" / "ขั้นที่ 2" + เรียงบนลงล่างตามลำดับที่ทำจริง
   อ่านง่ายกว่าย่อหน้าที่บรรยายลำดับ
3. **บอกผลลัพธ์ที่ปุ่มนั้นจะทำ ตรงปุ่ม** — "บันทึกแล้วต้นทุนจะเพิ่ม ฿X กำไรจะลด ฿Y"
   มีค่ากว่าย่อหน้าอธิบายว่าปุ่มนี้คืออะไร
4. **1 หน้าจอ = 1 ประโยคอธิบายพอ** ที่เหลือให้ตัวเลข/ป้าย/ลำดับพูดแทน

## 🔴 กับดักที่แพงกว่า: กล่องอธิบายที่ค้างอยู่ = กลายเป็นข้อมูลผิด

รอบนี้เจอว่ากล่องเขียว "ขั้นต่อไป — ตัดจ่ายค่าตู้ …" **ขัดกับหัวข้อของหน้าตัวเอง** ที่เขียนว่า
"ระบบตัดจ่ายเป็นบิล ไม่ใช่ทั้งตู้" — เพราะโมเดลเปลี่ยนไปแล้ว (mig 0273) แต่กล่องอธิบายไม่มีใครตามลบ
และมันยังชี้ไปทะเบียนเก่าที่ **prod 0 แถว ไม่เคยใช้เลย**

> กล่องอธิบายไม่มี type checker ไม่มีเทส — มันจึงเน่าเงียบกว่าโค้ดทุกชนิด
> **เปลี่ยนโมเดลเมื่อไร ให้ grep คำอธิบายที่พูดถึงโมเดลเก่าทันที**

เช็คลิสต์ตอนแตะหน้าไหนก็ตาม: คำอธิบายบนหน้านี้ยัง **จริง** อยู่ไหม · ยังชี้ไปที่ที่ยังใช้อยู่ไหม ·
ขัดกับหัวข้อ/ปุ่มบนหน้าเดียวกันไหม.
