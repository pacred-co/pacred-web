# ปุ่มบนจอ ห้ามเป็นคนตัดสิน "ฐาน" ของเงิน — และด่านต้องอยู่ตรงรอยต่อที่มีผลจริง

> 2026-07-23 · owner: *"ตู้นี้ทำไมไปโชว์ −เป็นแสนบาทเลยครับ แต่พอกดข้างใน +สามหมื่นห้า
> ทันยังไงกันแน่ครับ ... พวกการแสดงผลพวกนี้สำคัญมากนะครับ ทุกข้อมูลทุกคนเชื่อ และเอาไป
> ทำงานจริงนะครับ"* → แล้วต่อด้วย *"ด่านวางผิดที่นะครับ ด่านใส่ที่อยู่ต้องกันไว้ตรงก่อนส่งไป
> รอชำระต่างหากครับ เพราะกระทบเรื่องเงิน กับ เก็บตังออกเอกสารครับ"*

หนึ่งหน้าจอโชว์กำไรตู้ **−330,786** อีกหน้าโชว์ **+35,583** ของตู้เดียวกัน. เบื้องหลังมี 3 บทเรียน
ที่คนละเรื่องกันแต่มาจบที่ปัญหาเดียว.

## 1. ปุ่ม toggle บนจอ กลายเป็นคนตัดสินหน่วยของเงิน

```ts
// actions/admin/report-cnt-detail.ts — โมดัล "ตั้งค่าต้นทุนตู้"
const basis = VALID_WH.has(wh) ? costBasisMode(wh) : mode;
//                                                   ^^^^ ปุ่ม cbm/weight ที่แอดมินกดบนจอ
```

กฎจริงคือ **ขนส่งเป็นคนกำหนดฐาน** (Sang/MX คิดตามน้ำหนัก · ที่เหลือคิดตามคิว). โค้ดนี้เคารพกฎนั้น
**เฉพาะเมื่อรู้จักโกดัง** — พอเจอแถวที่ `fwarehousename` ว่าง มันตกไปใช้**ความตั้งใจของคนกด**แทน

`admin_audit_log` เก็บหลักฐานไว้ครบ:
```
2026-07-21  report_cnt.custom_rate  GZE260720-1  {"mode":"weight","rates":{"p1":4700},"row_count":83}
```
80 แถวที่ `wh='8'` เข้าทางขนส่ง → คิว → ถูก · 4 แถวที่ `wh=''` เข้าทางปุ่ม → **78.50 กก. × 4,700 =
฿368,950 เป๊ะ**. และมันไม่ใช่อุบัติเหตุครั้งเดียว — **12 จาก 105 ครั้ง** ที่เคยกดปุ่มนี้ เลือก `weight`

**กฎที่ได้:** ฐาน/หน่วยของการคิดเงินต้องมาจาก **ข้อเท็จจริงที่วัดได้** (ขนส่ง · ประเภทสินค้า) เท่านั้น
ห้ามให้ intent ของ caller (ปุ่ม · flag · พารามิเตอร์) เป็นตัวตัดสิน. ถ้าอยากเก็บว่าคนกดอะไร → log ไว้
เป็นเจตนา แต่อย่าเอาไปคูณเงิน

**พิสูจน์ว่ากฎถูก ก่อนถอดสิทธิ์ปุ่ม:** เรททั้งระบบ 42 ตู้ = 2,500-4,700 · tb_settings 2,400-6,500
= **ต่อคิวล้วน ไม่มีเรทต่อกิโลอยู่เลย** ⇒ คูณเรทต่อคิวด้วยกิโล = หน่วยผิดเสมอ ไม่ใช่ทางเลือก

**ด่านที่ใส่กันซ้ำ ต้องยึดข้อเท็จจริง ไม่ใช่คำประกาศ:** `checkCostWritePlausible()` เทียบกับ
**ขนส่งของแถว** ไม่ใช่ basis ที่ผู้เขียนบอกมา → คนเขียนที่*บอกว่า* cbm แต่ส่งเลขที่มาจากน้ำหนัก
ก็ยังโดนจับ. (ด่านเดิมเป็นเพดานต่อตู้ 5M → 391k ลอดสบาย — เพดานหลวมๆ ไม่ใช่ด่าน)

## 2. เลขเดียวกัน 2 เครื่องคิด = วันหนึ่งต้องต่างกัน

หน้าลิสต์ Σ `fcosttotalprice` ที่เก็บไว้ · หน้ารายละเอียดคิดสด (คิว × เรท). ตราบใดที่ค่าที่เก็บถูก
สองจอตรงกันพอดี — **จนกระทั่งมีค่าขยะเข้ามาแถวเดียว** แล้วสองจอก็เถียงกันทันที และไม่มีใครรู้ว่า
จอไหนพูดจริง

แก้ที่โครงสร้าง: `container-cost-engine.ts` ตัวเดียว ทั้งสองจอเรียก → **ต่างกันไม่ได้เชิงโครงสร้าง**
(replay 30 ตู้บน prod: 29 ตู้ drift 0 = ไม่ regress ของที่ถูกอยู่แล้ว)

ผลพลอยได้ที่เจอเพราะรวมเครื่องคิด: แถว `fcosttotalprice = 0` ที่ไม่เคยตั้งราคา — จอหนึ่งนับ 0
อีกจอคิดสดได้ 99.88. เครื่องเดียวกันทำให้ค่าที่เงียบแบบนี้โผล่มาเอง

## 3. ด่านต้องอยู่ตรง "รอยต่อที่มีผล" ไม่ใช่ตรงปลายทางกายภาพ

พอปลดล็อกให้งานที่ยังไม่รู้ที่อยู่เข้าระบบได้ (owner: *"ให้งานที่ถูกต้อง มันเข้าไปก่อนเถอะครับ"*)
ผมใส่ด่านกันไว้ที่ **มอบงานคนขับ** — ด้วยเหตุผลว่า "คนขับส่งของไปที่ว่างเปล่าไม่ได้"

owner แก้ให้: ด่านต้องอยู่**ก่อนส่งไปรอชำระเงิน** เพราะที่อยู่คือตัวกำหนด **ค่าส่งไทยที่เข้าบิล** และ
**ที่อยู่ที่พิมพ์บนเอกสารเงิน** — กว่าจะถึงคนขับ บิลออกไปแล้ว เอกสารผิดไปแล้ว

**กฎที่ได้:** วางด่านที่รอยต่อซึ่ง**ผลเสียเกิดขึ้นจริง** (เงินถูกคิด · เอกสารถูกออก) ไม่ใช่ที่ขั้นตอน
กายภาพสุดท้ายที่นึกออกง่ายที่สุด. ถามว่า *"ถ้าปล่อยผ่านตรงนี้ อะไรที่แก้คืนไม่ได้จะเกิดขึ้น"*

**หลักการที่ตกผลึก: ขาเข้าผ่อน · ขาเงินเข้ม** — ให้ข้อมูลเข้าระบบได้ง่าย (ไม่งั้นงานค้างนอกระบบ
= มองไม่เห็น) แต่ห้ามเดินข้ามรอยต่อที่ผูกกับเงินจนกว่าจะครบ

**และ sweep ให้ครบทุกทางเข้า:** รอยต่อ 4→5 มี **7 ทาง** รวมทางที่ไม่ผ่าน 5 เลย —
`adminMarkForwarderCredit` เขียน `fstatus='6'` ตรงๆ พร้อมสร้างหนี้ใน `tb_credit` = ข้ามรอยต่อ
ทั้งดุ้น. ด่านที่ปิดแค่ทางหลักคือด่านที่มีประตูหลัง

## 4. วัดรัศมีระเบิดของด่าน กับ prod ก่อนปล่อย

ด่านที่อยู่รอบแรกผมตั้งว่า *"ต้องมีจังหวัด **และ** รหัสไปรษณีย์"* — ฟังดูสมเหตุผล
พอ query prod: **จะกัน 18 แถว ซึ่ง 15 แถวมีจังหวัดแล้วแค่ไม่มี zip และ 7 แถวมอบคนขับสำเร็จไปแล้ว**

ด่านที่เข้มเกินไปคือบั๊กที่แต่งตัวเป็นความปลอดภัย. ผ่อนเหลือ *"ไม่มีทั้งจังหวัดและบ้านเลขที่"* → กัน
เหลือ 2 แถวตรงรอยต่อจริง — พอดีกับที่ควรกัน

**ก่อน ship ด่านทุกตัว: `SELECT count(*)` ว่ามันจะกันอะไรบ้างบนข้อมูลจริง** ถ้าตัวเลขบานเกินคาด
แปลว่าเกณฑ์ผิด ไม่ใช่ข้อมูลผิด

## 5. ป้ายที่ขัดกับราคา แย่กว่าไม่มีป้าย

MOMO ส่งรหัส `special`/`control` มา เราแปลเป็นป้าย **"น้ำยา"/"ควบคุม"** — คำที่**ไม่มีอยู่ใน
ระบบประเภทสินค้าของเราเลย** (มีแค่ ทั่วไป/มอก./อย./พิเศษ). แถวที่จอเขียน "น้ำยา" ระบบคิดราคา
เป็น *พิเศษ* → พนักงานอ่านป้ายแล้วเข้าใจเรทผิด

แก้ที่ราก: ป้าย **derive จากแผนที่ตัวเดียวกับที่ใช้คิดราคา** (`momoTypeToProductType`) → ป้ายกับ
เรท drift กันไม่ได้ ไม่มี map ป้ายตัวที่สองให้หลุดอีก

และตอน owner ต้องเคาะว่า `control`/`special` ควรเป็นประเภทไหน — **กางตัวเลขเงินให้ดูก่อน**
ไม่ใช่ถามลอยๆ: control ย้าย = ฿0 · special ย้าย = PR628 จ่ายเพิ่ม ฿12,414 · และทั้งระบบมี
735 รายที่ตั้งเรท อย.≠พิเศษ ต่างได้ถึง ฿11,000/คิว. เจ้าของตัดสินใจได้ใน 1 นาทีเพราะเห็นเงิน

## 6. เทสที่ไม่ได้ลงทะเบียน = เทสที่ไม่เคยรัน

`lib/forwarder/resolve-box-dims.test.ts` ถูกเขียนขึ้นพร้อมฟีเจอร์ ผ่าน 15 assertions ตอนเขียน
แล้ว**ไม่เคยถูกใส่ใน `package.json`** → ไม่เคยรันใน gate อีกเลย. เจอตอน merge เพราะบังเอิญ grep

เขียนเทสเสร็จ ตรวจว่ามันอยู่ใน **ทั้ง `test` และ `test:unit`** — ไม่งั้นมันคือไฟล์ที่ทำให้เราสบายใจเฉยๆ
