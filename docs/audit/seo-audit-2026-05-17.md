## §3 — Pre-existing observations (flag for post-launch)

### `[post-launch]` — Two `<h1>` per service page

Every service/landing page renders **two** `<h1>` elements:
1. `BookingHero` H1 (inside `BookingCalculator`) — dynamically reads the active tab + sea mode and shows the matching headline (e.g. "นำเข้าสินค้าจากจีน LCL แชร์ตู้" when sea-mode + LCL tab active)
2. Page-level hero H1 (the one I edited this pass)

Multiple H1s on a page are not a hard SEO penalty under modern Google rules — search picks the most relevant — but it dilutes the primary keyword signal and is against HTML5 outline guidance. The launch-readiness audit at `docs/research/prelaunch-frontend-2026-05-17.md` claimed "exactly one `<h1>` per page" but that was a code-level skim — runtime DOM has two.

**Fix path (deferred until after Monday launch):**
- Change `components/booking/BookingHero.tsx` L57 `<h1>` → `<h2>` *and* on the home page, ensure there is a page-level H1 (the home page currently relies on `BookingHero` for its only H1)
- Add a `headingLevel?: "h1" | "h2"` prop to `BookingHero` so the home page keeps an H1 while service pages use H2

Not for tomorrow's launch — would touch every page and need a verify pass. Risk > reward T-1.

### `[post-launch]` — "Soon" service cards self-link

5 grid cards on `/services` carry `status: "soon"` and `href: "/services"` (a self-link). Rendered as `aria-disabled` + `cursor-default` so not clickable — not a bug.
