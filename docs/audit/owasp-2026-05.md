## A01 — Broken Access Control

Pacred's access control sits at three independent layers, each defence-in-depth:

1. **Middleware (`proxy.ts`)** — refreshes Supabase session cookies on every
   request; passes auth + user ID downstream to RSC + route handlers.
2. **App layer (`requireAuth()` / `requireAdmin()` / `requireGuest()`)** — at
   the layout / page / server-action boundary. Three groups:
   - `(auth)` calls `requireGuest()` — redirects logged-in users away
   - `(protected)` calls `requireAuth()` — redirects guests + incomplete profiles
   - `(admin)` calls `requireAdmin([roles])` — variadic — 404s non-admins
3. **Database (Postgres RLS)** — every customer-table policy gates by
   `auth.uid()` ownership; admin policies use the `is_admin()` SECURITY DEFINER
   function (ADR-0002). Means even a compromised app-layer can't escalate
   beyond what RLS allows.

**Spot checks performed:**

- `actions/admin/*` — every mutation wrapped in `withAdmin([role], async ({adminId}) => ...)` per `conventions.md` §4 contract. `admin_id` recorded in audit log via `logAdminAction()`.
- `actions/wallet.ts::createDeposit` — uses customer-context Supabase client (`createClient()`), not admin client; customer can only insert deposits for their own `profile_id` via RLS policy `wallet_transactions_self_insert`.
- `app/[locale]/(admin)/admin/drivers/[id]/page.tsx` + `app/[locale]/(admin)/admin/drivers/page.tsx` — gain `requireAdmin(["ops"])` per P-18-followup-rbac (commit `8bd04b7`) — page-level + sidebar gate.

**Open gaps:**
- Admin RBAC is currently coarse — 4 roles (`super`, `ops`, `accounting`, `sales_admin`). Some routes assert `is_admin()` without further role check. Granular permission per-action is in Phase G2+ roadmap, not blocking for beta.

## A02 — Cryptographic Failures

- **OTP codes**: stored as `code_hash` (SHA-256 + server pepper `OTP_PEPPER`). Never log raw codes. TTL 5 min. Per-phone rate limit 3/hour via `otp_codes` DB-backed counter.
- **Passwords**: Supabase Auth handles bcrypt via its hosted Postgres function — Pacred never sees plaintext after `signInWithPassword()`.
- **Cookies**: `@supabase/ssr` issues httpOnly cookies with `SameSite=Lax` (default). The custom `pacred_vid` visitor cookie (`proxy.ts`) is `httpOnly: false` intentionally so client analytics can read; it carries only a UUID with zero PII.
- **TLS**: enforced by Vercel + HSTS header in `next.config.ts` (D-14).
- **SUPABASE_SERVICE_ROLE_KEY**: server-only via `import "server-only"` in `lib/supabase/admin.ts` — webpack throws if accidentally imported into a client component.

**Open gaps:**
- `OTP_PEPPER` rotation cadence not documented. Recommend: rotate quarterly; on rotation, set `OTP_PEPPER_NEXT` to new value, accept either, then flip after 1 hour (OTP TTL window).

## A03 — Injection (SQL, XSS, OS command)

**SQL:**
- All DB access via `@supabase/supabase-js` PostgREST client — parameterised; no string-concat SQL anywhere in `actions/` / `lib/`.
- Server actions take Zod-validated input — strict schemas reject malformed shapes before they reach the DB call.

**XSS:**
- React auto-escapes children. `dangerouslySetInnerHTML` audited for the only intentional uses:
  - `app/layout.tsx:94` — theme init script (static THEME_INIT_SCRIPT constant; no user input)
  - `components/analytics/gtm-script.tsx:21` — GTM loader (env-var interpolated into static template; `NEXT_PUBLIC_GTM_ID` is set by เดฟ in Vercel, not user-controlled)
  - `components/analytics/clarity-script.tsx:19` — Clarity loader (same pattern)
- All three places interpolate config we control, not customer input.
- ✅ **No customer input** flows into `dangerouslySetInnerHTML` anywhere.

**OS command:**
- N/A — no `child_process.exec` / shell-out in the codebase. `pnpm exec` invocations are dev-only via package.json scripts.

## A04 — Insecure Design

**Auth flow:**
- Multi-step juristic register defers all server calls to step 3 (per `PACRED-SECOND-BRAIN.md` §6.2) — avoids session-leak race.
- OTP: send-rate + verify-rate both bounded (3/hour, 5 attempts → forced burn).
- Phone change requires fresh OTP (P-3) — atomic update of both `auth.phone` + `profiles.phone` so half-update can't leak.

**Money flow:**
- Customer deposit slip → admin approve → wallet credit (no auto-credit on unverified slip).
- Wallet bucket recompute via DB trigger `wallet_recompute_balance` — single source of truth; no risk of UI/server drift.
- Order placement deducts wallet via the same trigger; concurrent orders against same wallet serialise via row-lock.

**Rate limiting:**
- Per-endpoint via `lib/rate-limit.ts` (D-12): signup 5/h/IP, login 10/h/IP, passwordReset 5/h/IP, contact 5/h/IP, generic 30/min.
- OTP-specific via DB counter (audit trail bonus).
- Memory-fallback in dev; Upstash Redis in prod (env vars optional — fallback used until set, with documented multi-instance leak risk).

## A05 — Security Misconfiguration

**Headers (`next.config.ts`, D-14):**
- `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` ✓
- `X-Frame-Options: DENY` ✓
- `X-Content-Type-Options: nosniff` ✓
- `Referrer-Policy: strict-origin-when-cross-origin` ✓
- `Permissions-Policy: camera=(), microphone=(), geolocation=()` ✓
- `Content-Security-Policy`: covers most directives. Notes below.

**CSP gap (🟡):**
- `script-src` includes `'unsafe-inline'` to accommodate Next.js' inline scripts + the theme-init script in layout.tsx. Common in Next App Router projects but warrants a follow-up to migrate to nonce-based CSP per Next 16 docs.
- `connect-src 'self' https: wss:` permits any HTTPS connection — broad. Acceptable for analytics + Supabase + LINE OA, but could tighten to explicit allowlist for stricter posture.

**PROMPTPAY_ID hard-fail (🟡):**
- `lib/promptpay.ts:21-25` throws if `PROMPTPAY_ID` unset. Customer hitting `/wallet/deposit` sees an uncaught error in prod. Should degrade to a friendly notice. Track as `D-7-followup-promptpay-degrade`.

**File upload:**
- `lib/file-validation.ts` (D-15) validates magic bytes server-side (not just MIME header). Wired in `actions/wallet.ts::createDeposit` and forwarder cover upload.
- Max 10 MB per upload. Storage buckets RLS-scoped per user.

## A06 — Vulnerable & Outdated Components

`pnpm audit` (the built-in security check, not our docs/env audit) reports:

| Advisory | Severity | Path | Status |
|---|---|---|---|
| postcss < 8.5.10 (GHSA-qx2v-qp2m-jg93) | Moderate | `>@sentry/nextjs>next>postcss` + `>next>postcss` + `>next-intl>next>postcss` | Open — Next 16 ships its own pinned postcss. Wait for Next 16.2.7+ patch release |

**Risk assessment:** The postcss vulnerability is a regex-DoS in source-map parsing during CSS build. Only triggered with attacker-controlled CSS at build time — not exploitable from runtime requests. Patch when Next ships a 16.2.x update that bumps postcss; track via Renovate / Dependabot.

**Lockfile hygiene:**
- `pnpm install --frozen-lockfile` required in CI + dev — prevents lockfile drift.
- `pnpm-workspace.yaml` explicit `allowBuilds` list (sharp, @parcel/watcher, @swc/core, @sentry/cli, unrs-resolver, esbuild) — controls which postinstall scripts can run.

## A07 — Identification & Authentication Failures

**Sign-up:**
- Phone+password (personal) or 3-step juristic with mandatory OTP.
- Email optional (`profiles.email`); profile starts `status='incomplete'` if OAuth + missing fields, forcing `/complete-profile`.

**Sign-in:**
- Identifier auto-detects: phone / email / member_code → resolves to user.
- Rate-limited 10/hour/IP.
- (Future) hCaptcha invisible on signup + forgot-password — code wired (D-13), env unset → no-op in dev, fails-closed in prod.

**Session:**
- JWT cookies (Supabase issued). Refresh on every request in `proxy.ts`. Sliding expiry per Supabase default.

**OTP:**
- Per-phone rate limit + per-OTP attempt limit (≥5 → force-burn). TTL 5 min.
- `OTP_BYPASS=true` in dev — must be `false` in prod (pre-launch checklist).

**Forgot-password:**
- P-2 flow: phone/email → OTP/email link → reset.
- Rate-limited 5/hour/IP via `checkRateLimit("passwordReset", ip)`.

**Open gaps:**
- 🟡 hCaptcha not yet active in prod (`NEXT_PUBLIC_HCAPTCHA_SITE_KEY` unset). Code is wired; just needs the K-13 / DV-1 activation.
- 🟢 LINE Login OAuth is a stub — disabled UI, not a real attack surface.

## A08 — Software & Data Integrity Failures

- CI workflow (.github/workflows/ci.yml) verifies lint + tsc + tests + audits on every PR. No untrusted CI runners.
- pnpm lockfile + `--frozen-lockfile` reject any dep version drift.
- `@react-pdf/renderer` is the only direct content-rendering library that handles complex documents; runs server-side under our control.
- Webhook receivers (future LINE OA bot) will verify signatures via `LINE_CHANNEL_SECRET` — not yet implemented, no public surface.
- Vercel cron endpoints (`/api/cron/*`) require `x-vercel-cron` header OR `Authorization: Bearer ${CRON_SECRET}` (D-17). Without `CRON_SECRET` set in prod, the endpoints fail closed.

## A10 — Server-Side Request Forgery (SSRF)

All outbound HTTP from server-side code targets a **fixed allowlist** of vendors:

| Endpoint | Service | Notes |
|---|---|---|
| `tamit-cloud.com/api-product` | China search detail | Env-var-gated; demo-mode if unset |
| `tam-i-t.com/api/convert-link-china` | China search cache | Same |
| `akucargo.com/api3/api-2022` | China search keyword | Same |
| `laonet.online/index.php` | China search image | Same |
| `api.line.me/v2/bot/message/push` | LINE Messaging API | Hard-coded host in `lib/notifications/index.ts` |
| `api.hcaptcha.com/siteverify` | hCaptcha verify | Hard-coded host in `lib/hcaptcha.ts` |
| `*.upstash.io` | Upstash REST | Env-var-gated; in-memory fallback when unset |
| `*.ingest.sentry.io` | Sentry tunnel | Via `/api/monitoring` proxy route |
| `opendata.dbd.go.th` | DBD tax-ID lookup | Hard-coded in `app/api/dbd/[taxId]/route.ts` |

**No user-controlled URLs** are fetched server-side. The closest case is `/api/china-search/` which takes user-pasted Taobao / 1688 URLs — but those go through `lib/china-search/extract-product-id.ts` which **parses** the URL for a product ID, then constructs requests against the hard-coded vendor base URL with only the product ID as a parameter. The user-pasted URL itself is never fetched directly.

**SSRF posture:** clean.
