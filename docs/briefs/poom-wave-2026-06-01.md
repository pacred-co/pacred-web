## 0. The shape of your lane (TL;DR)

The **money-document + commission** layer is the *most mature, most-faithful* part of the whole port —
AND it carries the biggest pile of **dead rebuilt twins** sitting next to the live legacy tables. Three truths:

1. **Receipts + WHT + combine-bill = faithfully ported and LIVE.** `tb_receipt` (13.8k) · `tb_bill` (10.6k) ·
   doc-no minter (`FRC/FRG{yyMM}-{NNNNN}`) · auto-issue-on-payment · a WHT/VAT engine that **exceeds** legacy.
   The PEAK-style 7-tab receipts explorer already exists (`/admin/accounting/receipts`). **No gap — your base.**
2. **Sales-rep commission earn→withdraw = faithfully ported and LIVE** on `tb_user_sales` (earn-trigger
   FIRES on real delivery — the prior "contested #6" is **RESOLVED**). The admin pay-out side
   (`/admin/sales-payouts`) is **already repointed** to faithful `tb_user_sales_admin_pay`.
3. **BUT 2 older admin pages are Potemkin** (`/admin/commissions`, `/admin/forwarder-sales` read dead
   rebuilt tables → 4,104 real earns invisible) **AND 2 legacy batch-payout systems are 0% ported**
   (`tb_withdraw_comm_sale_*` + `tb_withdraw_comm_interpreter_*` — 71 batches / 6,151 line-items of real
   history with no Pacred reader/writer).

The owner likes **PEAK** → your north-star deliverable is a **PEAK-like documents + ledger module**
(invoice→receipt→tax-doc lifecycle · debtor/AR-aging · period close · e-Tax readiness), built ON the live
`tb_*` data you already have.

**⚠️ Legacy "ระบบบัญชี" = 95% UI stub** (per `legacy-accounting-reality-2026-05-30.md`): 3 dropdowns × ~50
sub-screens but almost every `?page=*` has NO handler. The ONLY thing that truly worked in legacy = the
`ใบเสร็จรับเงิน` (receipt, *not* tax-invoice) doc + the single `callPriceUser` button. So **do NOT** chase
"faithful 1:1" of the legacy accounting menubar — there's nothing behind it. PEAK-style is a *Pacred
improvement* on top of the one real doc (receipt) + the WHT engine. This is Phase-C-flavoured but
owner-requested; build it as a clean new module, don't pretend it's a port.

**Coordination with เดฟ (shared money tables):** เดฟ owns the wallet ledger + customer pay-paths
(`tb_wallet`/`tb_wallet_hs`/`tb_payment`/`tb_credit`/`tb_cash_back`) and is running the "trust sweep"

## 2. Port the 2 legacy batch-payout systems (the biggest faithful-port gap)

### Goal
Build the admin-driven **monthly batch payout** workflows that pay reps + interpreters in bulk. 6,151
line-items + 71 batches of real payout history are currently invisible & unmanageable in Pacred.

### Tables — LIVE legacy (real data) · ZERO Pacred code touches them (`grep tb_withdraw_comm` = 0 files)
| Table | Rows | Purpose |
|---|---:|---|
| `tb_withdraw_comm_sale_h` | 25 | Sales-rep batch payout header (amount · commbefore · withholding · status 1/2/3 · adminid=rep · bank fields · imagesslip · title) |
| `tb_withdraw_comm_sale_item` | 3,204 | `fid` (→`tb_forwarder.id`) · `wcsid` (→header) |
| `tb_withdraw_comm_interpreter_h` | 46 | Interpreter (ล่าม) batch payout header (same shape; adminid=interpreter) |
| `tb_withdraw_comm_interpreter_item` | 2,947 | `hno` (→`tb_header_order.hNo`) · `diffyaun` (yuan margin) · `wciid` (→header) |
| `tb_set_comm_interpreter` | 15 | Per-interpreter commission **%** setting (`percom` · `adminid`) — **no Pacred ref** either |
| `tb_account_pcs` | 98 | Company bank accounts (the "pay-FROM" source for both flows) |

### Legacy logic (from `withdraw-commission-{sale,interpreter}.php` + their `include/pages/.../{home,add}.php`)
- **Sales-rep batch:** admin selects unpaid forwarders for a rep → commission = **1% of `fTotalPriceNetAll`**
  (`add.php:333`) summed from PAID `tb_wallet_hs` rows → create batch header w/ WHT + bank acct (from
  `tb_account_pcs`) → status 1 → upload slip → status 2. RBAC: CEO/Manager/Accounting/ITDT/SaleCargo/SalesAll
  (reps see only their own).
- **Interpreter batch:** same shape but commission = per-order **yuan margin × `tb_set_comm_interpreter.perCom`**
  from `tb_header_order` (yuan orders).
- WHT deducted on the batch (the `withholding` column) — feed it through `lib/tax/wht.ts` (service 3%, or 1%
  on e-WHT remit per the lib's note).

### Existing code: **MISSING** (0 files). Build new.
- New actions: `actions/admin/withdraw-comm-sale.ts` + `actions/admin/withdraw-comm-interpreter.ts` (mirror
  `sales-payouts-tb.ts` structure: list-pending · detail · add-batch · upload-slip · server-recompute).
- New pages under `/admin/accounting/withdraw/` (already a dir) or `/admin/commissions/batch-{sale,interpreter}/`.
- New `tb_set_comm_interpreter` editor (admin config %, like the rate editors).
- **Money-safe rules to copy from `commissions-tb.ts`:** NEVER trust the client amount — recompute
  server-side from live `tb_forwarder`/`tb_header_order` over re-validated unpaid rows; dedup-guard before
  INSERT (a `fid`/`hno` can't be in two batches); status-guard on pay (`AND status=2` before flipping to 3,
  prevents double-pay — same guard `sales-payouts-tb.ts` uses).

### Acceptance (§0c · money-safe)
- List page shows the 25 + 46 real historical batches (read-back works); detail page fans out to the
  forwarder/order line-items with the per-order amounts. Create-batch dry-run computes 1% (sale) / margin×%
  (interpreter) matching a hand-checked legacy row. Slip upload flips status 2→3 with the double-pay guard.
  Click-through every action. All reads/writes via admin client (`tb_*` is RLS service-role-only).

**Effort: L · Priority: P1** (P0 if owner needs monthly payouts before launch — confirm with him).

## 5. Auto-commission accrual engine + unify the 3 commission SOTs (ADR)

### Goal
(a) Make commission accrual a **trigger** (like the earn-trigger) instead of lazy per-month SQL: on
`tb_wallet_hs` paid → write a typed accrual row → reps/interpreters see a live "owed" balance, admin batches
just confirm. (b) Kill the SOT split-brain: **THREE** commission models live in code —
`tb_user_sales` (canonical), the DEAD `commission_*` rebuilt stack, and the un-ported `tb_withdraw_comm_*`
legacy batch stack. Consolidate to ONE.

### The decision (write an ADR — extend `docs/decisions/0020-commission-sot.md` or a new 0023)
Recommend: **extend the `tb_user_sales*` family to cover batch + interpreter** (it already has earn +
payout + link tables and a server-recompute anti-tamper path), **port `tb_withdraw_comm_*` as the historical
batch reader** (§2), and **delete `commission_*`** entirely. Prevents the next dev wiring a 4th dead twin.

### Existing code
- Canonical: `tb_user_sales` family (`commissions-tb.ts`, `sales-payouts-tb.ts`, `earn-trigger-tb-user-sales.ts`).
- DEAD-to-delete: `commission_*` (`actions/admin/commissions.ts` + `actions/commissions.ts`).
- MISSING: the accrual trigger (today earn fires on delivery; batch comm is computed lazily in the selector).

### Acceptance
ADR merged + linked from `_MASTER-PLAN.md` §3. `grep commission_` returns only tombstones. Accrual trigger
writes a typed row on paid `tb_wallet_hs`; a rep's "owed" balance matches a hand-summed set of their unpaid
earns.

**Effort: M · Priority: P0** (the ADR is cheap and unblocks §1/§2 cleanly; the trigger is the heavier half — U4+U8).
