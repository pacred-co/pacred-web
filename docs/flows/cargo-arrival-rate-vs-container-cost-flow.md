# โฟลว์ "ของเข้ามา → ตั้งเรท vs ตรวจตู้/จ่ายค่าตู้" (ฝากนำเข้า/Cargo)

> ภูม 2026-06-30 — เคลียร์ข้อเถียงกับบัญชี: "ของเข้ามา โกดังยิงรับเข้า แล้วบัญชีตรวจตู้ก่อน หรือเซลตั้งเรทก่อน?"
> **คำตอบสั้น: หลังโกดังชั่ง/วัด แล้ว — เซลตั้งเรท(ขาย) กับ บัญชีจ่ายค่าตู้(ต้นทุน) เป็นคนละเส้น ทำขนานกันได้ ไม่ต้องรอกัน. สิ่งเดียวที่ต้องมาก่อนเสมอ = โกดังชั่ง/วัดคิว-น้ำหนักจริง.**

---

## มี 2 เส้นเงิน แยกกันสนิท (อย่าเอามาปนกัน)

| | เส้น A — **ขาย** (SELL) | เส้น B — **ต้นทุนตู้** (COST) |
|---|---|---|
| เงินไหลทางไหน | **ลูกค้า → Pacred** | **Pacred → ขนส่งจีน (MOMO)** |
| ใครทำ | เซล/แอดมิน ตั้งเรท → บัญชีรับเงิน | บัญชี ตรวจตู้ + จ่ายค่าตู้ |
| คิดจาก | เรทขายลูกค้า × คิว/น้ำหนัก | เรทต้นทุน × คิว/น้ำหนัก (ที่ MOMO เก็บเรา) |
| โผล่ที่ | สถานะ forwarder `fstatus` 5→6→7 + ใบวางบิล/ตรวจสลิป | หน้า `report-cnt` ปุ่ม "จ่ายเงินค่าตู้ + แนบสลิป" |
| บล็อกกันมั้ย | ❌ ไม่บล็อกกัน — ตั้งเรทไม่ต้องรอจ่ายค่าตู้ · จ่ายค่าตู้ไม่ต้องรอตั้งเรท |

**ทั้งคู่ต้องใช้ "คิว/น้ำหนักจริง" เป็นตัวคูณ → เลยต้องรอโกดังชั่ง/วัดก่อน. หลังจากนั้นทำพร้อมกันได้เลย.**

---

## ลำดับเวลาจริง (timeline)

```
1) ของถึงโกดังจีน        fstatus 2  ← MOMO/โกดังจีน
2) ส่งมาไทย              fstatus 3
3) 🟡 โกดังไทย "ยิงรับเข้า" + ชั่งน้ำหนัก/วัดคิว   fstatus 4 ถึงไทย   ← ★ จุดตั้งต้น (ต้องมาก่อน)
        │
        ├─ เส้น A (ขาย):  ระบบดึง "เรทขาย" ของลูกค้า × คิว → ได้ราคาขาย
        │                 → เด้งเป็น fstatus 5 "รอชำระเงิน"
        │                 → ออกใบวางบิล → ลูกค้าโอน → บัญชีตรวจสลิป (2 รอบ) → ตัดจ่าย
        │                 → fstatus 6 เตรียมส่ง → คนขับส่ง → fstatus 7 ส่งแล้ว
        │
        └─ เส้น B (ต้นทุน): ตู้ปิด → MOMO วางบิลค่าตู้ให้เรา
                          → บัญชีเข้า report-cnt "ตรวจตู้" + "จ่ายเงินค่าตู้ + แนบสลิป"
                          → ตู้ = "จ่ายแล้ว"
```

ข้อ 3 (โกดังชั่ง) คือ **ต้องมาก่อน** เพราะทั้ง 2 เส้นเอาคิว/น้ำหนักไปคูณ. แต่ A กับ B **หลังจากนั้นไม่มีลำดับบังคับ** — ใครพร้อมก่อนทำก่อน.

> หมายเหตุ MOMO: แถวที่ดึงมาจาก MOMO API จะมี **เรทขาย (frefrate) + คิว/น้ำหนัก ติดมาตั้งแต่ fstatus 2** อยู่แล้ว (ไม่ต้องรอโกดังไทยชั่งซ้ำ ถ้า MOMO ส่งครบ) → ราคาขายคำนวณได้เลย. เคสที่โกดังไทยต้องชั่งเอง = ลูกค้าสร้างรายการเอง/ของที่ MOMO ไม่ได้ส่งน้ำหนักมา.

## ต่างจาก legacy (PCS Cargo) ยังไง

**โครงเหมือนกันเป๊ะ** — legacy ก็มี fstatus 1-7 + ตู้ (cnt) แยกเส้นต้นทุนเหมือนกัน. `report-cnt.php` ก็คิดกำไร = (ราคาขาย stored − ต้นทุน stored) เหมือนกัน. 2 เส้น A/B แยกกันเหมือนกัน.

**ที่ Pacred เพิ่ม/ต่าง:**
1. **MOMO auto-import** — legacy แอดมินคีย์เอง; Pacred ดึง API → เรทขาย+คิวติดมาตั้งแต่ fstatus 2.
2. **คำนวณสด (self-heal) บนหน้า report-cnt** — legacy ถ้าแถวยังไม่ตั้งเรท (`ftotalprice`=0) มันก็โชว์กำไรติดลบมั่วเหมือนกัน (legacy แค่ไม่ค่อยเจอเพราะคีย์ครบก่อน). Pacred แก้ให้ **derive ราคาขาย/ต้นทุน สดจากเรท × คิว** เมื่อยังไม่ตั้งราคา+ตู้ยังไม่จ่าย → กำไรไม่ติดลบมั่ว (แก้ 2026-06-30). ตู้ที่จ่ายแล้วคงค่า stored (ล็อก).

---

## สรุปให้บัญชี (1 บรรทัด)

> โกดังชั่ง/วัดก่อน → จากนั้น **เซลตั้งเรท(ขาย)** กับ **บัญชีจ่ายค่าตู้(ต้นทุน)** เป็นคนละเรื่อง ทำพร้อมกันได้ ไม่มีใครต้องรอใคร. กำไรตู้ที่เห็น = ราคาขาย(เรทลูกค้า×คิว) − ต้นทุน(เรทตู้×คิว).
