# `supabase/bundle/` — Prod-fresh schema bundle

> **Generated 2026-05-20 ค่ำ** (ภูม) per the move-to-supabase-prod-only
> directive. ก๊อต took the dev project for other work; this folder is the
> portable single-file SQL bundle ภูม can re-run against any fresh Supabase
> project to reproduce the exact production schema.

## What's here

| File | Purpose |
|---|---|
| `prod-fresh.sql` | Concatenation of every migration in `supabase/migrations/0*.sql`, in numeric order, with per-file delimiter comments. ~21,900 lines / 891 KB. |
| `README.md` | This file. |

## When to use this

**Use it** when you need to recreate the schema on a brand-new Supabase
project — e.g. spinning up a new dev/staging, recovering from a deleted
project, or onboarding a new environment.

**DO NOT use it** on an existing project that already has migrations
applied. The bundle is not idempotent — it expects an empty DB. Re-running
against a populated DB will conflict on CREATE TABLE / CREATE TYPE / etc.

For ongoing schema changes use the normal flow: add a new numbered
migration to `supabase/migrations/`, apply via `supabase db push` (or
psql), and regenerate this bundle (see § "Regenerating").

## How to run

### Option 1 — Supabase SQL editor (recommended for one-shot)

1. Create the new Supabase project.
2. Open SQL editor.
3. Paste the entire contents of `prod-fresh.sql`.
4. Run. (Supabase SQL editor can handle the ~900 KB; it may take 30-60s.)

### Option 2 — `psql` (recommended for CI / scripted)

```bash
export PROD_DB_URL="postgresql://postgres:<password>@db.<ref>.supabase.co:5432/postgres"
psql "$PROD_DB_URL" -f supabase/bundle/prod-fresh.sql
```

### Option 3 — Supabase CLI

If you prefer the official migration tracking, run the migrations
individually (each filename gets recorded in the `supabase_migrations`
table — useful if you want future `supabase db push` to be a no-op):

```bash
supabase db push --db-url "$PROD_DB_URL"
```

The CLI runs each `supabase/migrations/0*.sql` file in order and stamps
the migration log. Result is identical to the bundle; only the
tracking differs.

## Order inside the bundle

| Range | Purpose | Lines |
|---|---|---|
| `0002`-`0080` | Pacred app schema (Phase 1-5 + Phase H + V/H1 RBAC + container spine + accounting + tax invoice + freight + HR + audit log + CSV imports + …). | ~12,800 |
| **`0081`-`0083`** | **Legacy PCS Cargo `tb_*` schema — 117 tables ported from `pcsc_main` (~8,898 customers · orders · wallets · ตู้ · forwarders · receipts · login hashes).** This is the D1 faithful-port foundation. | ~8,700 |
| `0084`-`0086` | Pacred extensions on top of legacy (booking documents · tax-invoice `credit_note_for` · work-item messages). | ~120 |
| `0087` | `migration_view` SECURITY INVOKER hardening. | ~10 |
| `0089` | `disbursement.kind` enum extension. | ~15 |

**Intentional gaps:** `0001`, `0065`, `0088` — these numbers are reserved
but the corresponding files don't exist (history: 0001 collapsed into 0002
during the initial setup; 0065 reserved for a wallet refactor that landed
in 0064 instead; 0088 reserved for the bundle generator itself but
unnecessary — the bundle lives in `supabase/bundle/` not as a migration).

## Legacy `tb_*` coverage audit

Generated by `diff`-ing `2026-05-18-1358-pcsc_main.sql` (the legacy MySQL
dump on ภูม's machine at `C:\Users\Admin\pcscargo\newdata\`) against
`0081_pcs_legacy_schema.sql`:

```
Legacy tables: 117
Our tables:    117
Missing from ours: (none)
Extra in ours:     (none)
```

Every legacy table is present. The `0081`/`0082`/`0083`/`0087` set
constitutes the **complete** legacy schema rewrite from MySQL to Postgres
(MySQL `varchar` → Postgres `character varying`, `tinyint` → `smallint`,
backtick identifiers → quoted lowercase, `enum` → text + CHECK, `bigint`
PKs preserved, indexes deferred to `0082` for parallelism).

**What this means in plain English:** running `prod-fresh.sql` against a
fresh Supabase gives you a database that is byte-for-byte schema-identical
to current production — Pacred app + legacy PCS Cargo overlay + all
extensions. Data must be loaded separately (per
`docs/runbook/pcs-data-migration.md` for the legacy customers/orders/etc.).

## Regenerating

When new migrations land, regenerate the bundle with:

```bash
cd <repo-root>
node -e "
const fs = require('fs'), path = require('path');
const dir = 'supabase/migrations';
const migs = fs.readdirSync(dir).filter(f => /^0\d+_.*\.sql\$/.test(f)).sort();
let out = '<header — see prod-fresh.sql for the canonical template>';
for (const name of migs) {
  out += '-- ╔════…\n-- ║ ' + name.padEnd(63) + '║\n-- ╚════…\n';
  out += fs.readFileSync(path.join(dir, name), 'utf8') + '\n\n';
}
fs.writeFileSync('supabase/bundle/prod-fresh.sql', out);
"
```

The bundle's first comment block (the header at top of `prod-fresh.sql`)
is the canonical template — copy the wording / `╔╗` box style from there.
