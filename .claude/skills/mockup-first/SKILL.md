# Mockup-First — ให้เห็นภาพก่อน แล้วค่อยเขียนโค้ดจริง

> **ทำไมต้องมีสกิลนี้ (owner 2026-07-24).** หัวร้านค้าหน้า ฝากสั่งซื้อ ทำสีแบรนด์แดงเต็มแถบ
> ผ่าน gate ครบ push ขึ้น main แล้ว owner เปิดดู → *"สีแดงแสบตาเกินไปหน่อยครับ"* →
> ต้องแก้ใหม่ทั้งรอบ. พอทำ **ปรีวิว HTML ให้ดูก่อน** owner ตอบทันทีว่า *"โอเคแล้วครับ"* และสั่งว่า
> *"ก่อนขึ้นงานโค้ดจริง ทำให้เราแต่งและเห็นภาพแบบนี้ก่อนเริ่มงานจริงดีเลยครับ เอาไปใส่ skills
> ไว้เลยครับ ให้ทุกคนได้ใช้ด้วย"*
>
> งานหน้าตา **ตัดสินด้วยตา ไม่ใช่ด้วย gate** — tsc/lint/build เขียวไม่ได้แปลว่าสวยหรือใช้งานได้.
> ปรีวิวใช้เวลา ~5 นาที · แก้หลัง push ใช้เวลาเป็นรอบ (แก้ → gate → push → รอ deploy → ถามใหม่).

## RULE 0 — งานหน้าตา = ปรีวิวก่อนเสมอ

| งานแบบไหน | ต้องปรีวิวก่อนไหม |
|---|---|
| สี · gradient · ฟอนต์ · spacing · การ์ด/แถบ/ป้าย | ✅ **ต้อง** |
| หัวตาราง · layout ใหม่ · จัดกลุ่มใหม่ | ✅ **ต้อง** |
| เอกสารพิมพ์ (ใบเสร็จ/ฟอร์ม) | ✅ **ต้อง** (+ `print-verify`) |
| แก้ตรรกะ/เงิน/สถานะ โดยหน้าตาไม่เปลี่ยน | ❌ ไม่ต้อง |
| เปลี่ยนคำ 1-2 คำ | ❌ ไม่ต้อง |

**ถ้าเข้าถึงหน้าจริงไม่ได้ (ต้อง login แต่กรอกรหัสผ่านไม่ได้) → ยิ่งต้องปรีวิว** เพราะ
นั่นเป็นทางเดียวที่จะให้ owner เห็นก่อนตัดสินใจ

## 4 ขั้นตอน

### 1. ดึงของจริงมาก่อน — โทเค็นสีจริง + ข้อมูลจริง

ปรีวิวที่ใช้สีมั่วหรือข้อมูลสมมติ = **ตัดสินใจบนของปลอม** owner อนุมัติแล้วของจริงออกมาคนละเรื่อง

```bash
# สีจริงจากธีม (ห้ามเดา hex เอง)
grep -n "color-primary" app/globals.css

# ข้อมูลจริงจาก prod (ตัวเลข/ชื่อ/ความยาวจริง — ชื่อร้านจีนยาวๆ ทำ layout พังได้)
SUPABASE_DB_PASSWORD=… node -e "…"   # หรือ scripts/ ที่มีอยู่
```

⚠️ ต้องใช้ **เคสสุดขั้ว** ด้วย: ชื่อยาวสุด · ตัวเลขเยอะสุด · ค่าที่เป็น 0 · ค่าที่ติดลบ

### 2. เขียน HTML เดี่ยวๆ ลง scratchpad

CSS ธรรมดา + ตัวแปรสีจริงที่ค้นได้ (ไม่ต้องพึ่ง Tailwind build):

```html
<!doctype html><meta charset="utf-8">
<style>
:root{--p50:#fff0f0;--p100:#ffe0e0;--p600:#b30000;--p800:#660000}  /* ← จาก globals.css */
body{font-family:"Prompt",-apple-system,sans-serif;background:#f1f5f9;padding:20px}
/* …ทำ 2-3 แบบให้เทียบกันในหน้าเดียวได้ยิ่งดี… */
</style>
```

**ทำ 2-3 ตัวเลือกในหน้าเดียว** ให้ owner ชี้ได้เลยว่าเอาอันไหน = จบใน 1 รอบ

### 3. render ให้เห็นจริง แล้วถ่ายรูป

```bash
cp <scratchpad>/preview.html public/_tmp-preview.html   # ต้องอยู่ในโปรเจกต์ถึงจะ render
```
```
preview_start {name:"pacred-1to1"} → navigate http://localhost:3000/_tmp-preview.html → screenshot
```
```bash
rm -f public/_tmp-preview.html    # 🔴 ลบทันที ห้ามหลุดเข้า repo
```

> ไฟล์นอกโฟลเดอร์โปรเจกต์จะ render เป็น snapshot นิ่งๆ (สีไม่ตรง) → **ต้อง copy เข้า `public/`
> ชั่วคราว** แล้วลบทิ้งทันทีในคำสั่งเดียวกันกับที่ถ่ายรูปเสร็จ · เช็ค `git status` ให้สะอาด

### 4. ให้ owner เคาะ แล้วค่อยเขียนโค้ดจริง

เอาผลลัพธ์ที่เห็นไปเล่าเป็นภาษาคน (ไม่ใช่ชื่อคลาส Tailwind) + เสนอทางเลือกที่ปรับได้
เช่น *"ถ้าอยากให้จางกว่านี้ หรือเปลี่ยนทิศ gradient บอกได้ครับ แก้บรรทัดเดียว"*

**เมื่อโค้ดจริงเสร็จ ต้องกลับมาตรวจว่าตรงกับปรีวิว** — ปรีวิวเป็น CSS มือ ของจริงเป็น Tailwind
คลาสที่แปลไม่ตรงจะได้ผลคนละอย่าง (`group-open:` · named group · dark mode)

## กับดักที่เจอจริง

**M-1 · เดา hex เอง** → ปรีวิวสวยแต่ของจริงคนละสี. อ่านจาก `app/globals.css` เสมอ

**M-2 · ใช้ข้อมูลสมมติ** → "ร้าน A" 5 ตัวอักษร ผ่านฉลุย แต่ `义乌市卿闻饰品厂` ของจริงดัน layout
พัง. ดึงจาก prod เสมอ + เอาเคสยาวสุด/เลขเยอะสุดมาด้วย

**M-3 · ลืมลบไฟล์ชั่วคราวใน `public/`** → หลุดขึ้น prod เป็นหน้าเปล่าที่ใครก็เปิดได้.
ถ่ายรูปเสร็จ `rm` ในคำสั่งเดียวกันเลย

**M-4 · ปรีวิวผ่าน แต่ Tailwind แปลไม่ตรง** → ก่อนใช้ variant ที่ไม่คุ้น
(`group-open/name:` · `marker:` · `@media print`) **เช็คว่าโปรเจกต์เคยใช้สำเร็จมาก่อนไหม**:
```bash
grep -rn "group-open" app components --include='*.tsx' | head -3
```
มีของเดิม = ใช้แพทเทินเดิม · ไม่มี = เสี่ยง ต้องพิสูจน์เพิ่ม

**M-5 · โชว์แต่โหมดสว่าง** → ระบบมี dark mode. ถ้าแตะสีพื้น/ตัวอักษร ต้องคิดคู่ `dark:` ด้วย
ตั้งแต่ตอนออกแบบ
